# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** For Sentry source-map upload from Vite, conditionally register `@sentry/vite-plugin` in `vite.config.ts` only when `SENTRY_AUTH_TOKEN` is present in the build env — `...(sentryAuthToken ? [sentryVitePlugin({...})] : [])`. This keeps local `npm run build` clean (no warning, no failed upload) while letting Vercel's build pick up the token. Pair with `build.sourcemap: 'hidden'` (emits maps but strips `//# sourceMappingURL` so browsers can't fetch them) and `sourcemaps: { filesToDeleteAfterUpload: ['./dist/**/*.map'] }` (deletes maps from `dist/` before Vercel publishes — maps live on Sentry only, not the public CDN). Sentry auth token needs `project:releases` scope.
- **[gotcha]** `@sentry/vite-plugin` config for this repo: `org: 'solo-23'`, `project: 'fog_and_frontier'`. Surface `VERCEL_GIT_COMMIT_SHA` + `VERCEL_ENV` from `process.env` (Vercel-injected at build) to the client as `VITE_VERCEL_GIT_COMMIT_SHA` / `VITE_VERCEL_ENV` via the `define` block in `vite.config.ts` — Vite does not auto-prefix non-`VITE_*` server-side env vars, so `initSentry()` cannot tag release/environment without this define-block bridge.
- **[gotcha]** A Vercel preview build for this repo can fail with `error TS2550: Property 'cause' does not exist on type 'Error'. Try changing the 'lib' compiler option to 'es2022' or later.` even though `tsconfig.api.json` already sets `target: es2023, lib: ["ES2023"]` and the same `tsc -b` succeeds locally with the same TypeScript version (`~6.0.2`). The Vercel build runs `npm run lint && tsc -b && vite build` from a clean install (no tsbuildinfo cache), so a clean local `tsc -b` is not equivalent. Don't trust a green local typecheck as proof that Vercel will build — push a small canary or check `vercel inspect --logs` after the first build attempt.
- **[gotcha]** If smoke's `fetchApiOk` retry exhausts all 4 attempts and still gets 500s on `/api/*`, it is NOT a Turso cold-start — cold-start recovers within ~5–15s, well inside the retry budget. Persistent 500s through the full budget mean the DB isn't reachable at all: deleted DB, stale/wrong Preview env vars, or `assertEnvMatchesUrl` in `api/_db.ts` is throwing. Before assuming flake, grep the smoke log for `[retry` and inspect Vercel function logs for the actual error — don't widen the retry window.
- **[pattern]** `turso db create <name>` (no `--group` flag) inside this project auto-attaches the new DB to the existing Vercel↔Turso Marketplace integration's group (`icfg-…-aws-us-east-1`), so the new DB is immediately visible in the Vercel Storage tab and `turso db list`. Verify with `turso db show <name> --url` — the URL should end in `-vercel-icfg-…`.
- **[gotcha]** `vercel env pull .env.<scope> --environment=<scope>` writes **empty strings** for variables marked sensitive in Vercel — for this project, `TURSO_DATABASE_URL` and `TURSO_AUTH_TOKEN` come back as empty strings. The `scripts/seed-preview-db.mjs` flow exits with `Missing prod TURSO_DATABASE_URL.` Workaround: fetch URL/token straight from `turso db show <name> --url` + `turso db tokens create <name>` and pass them as inline env vars to a small inline `node --input-type=module -e '...'` script.
- **[gotcha]** Long secrets (e.g. 267-character Turso JWT tokens) cannot be piped through `xargs -I {} vercel env add ... --value '{}'` — bash errors with `xargs: command line cannot be assembled, too long`. Pass the token as a shell variable to `--value "$TOKEN"` directly instead.
- **[pattern]** When sibling-branch leftovers appear as untracked in a new feature branch, stage your work explicitly by file name rather than `git add -A` / `git add .`. `git add README.md api/_log.ts api/_log.test.ts …` keeps the unrelated untracked files out of the commit.
- **[gotcha]** `vi.spyOn(console, 'error').mockImplementation(...)` in this repo trips `@typescript-eslint/no-unsafe-*` rules because `spy.mock.calls[0]?.[0]` is typed as `any`. Since the no-suppression policy forbids `eslint-disable` and the `any` propagates everywhere you read mock calls, prefer a hand-rolled capture helper that exposes a `string[]` array.
- **[gotcha]** `@typescript-eslint/no-redundant-type-constituents` flags `unknown | Promise<unknown>` (and any `unknown | X` union) — `unknown` already includes every type. For a handler type that can return sync-or-async, just write `(req, res) => unknown`. The wrapper can still `await handler(req, res)` — awaiting a non-Promise is a no-op.
- **[gotcha]** `@typescript-eslint/no-unnecessary-type-assertion` fires on `value as T`, assertion-at-declaration patterns (`const req = fakeReq() as VercelRequest & {...}`), `(import.meta.env.MODE as string)` (Vite already types MODE as string), AND on `as` casts inside inline object literals when the receiver already accepts that type (e.g. `[{...} as Exception]` when array element type is already `Exception`). Fix: declare a typed const first (`const exc: Exception = {...}`) and reference it; or move assertions to the assignment site.
- **[pattern]** Use `vi.stubEnv('NAME', 'value')` + `vi.unstubAllEnvs()` (in `afterEach`) to set Vite/process env in a test rather than direct `process.env.NAME = 'value'` assignment. To test a module that reads env at import time AND calls `Sentry.init`, combine `vi.hoisted(() => ({ init: vi.fn() }))` + `vi.mock('@sentry/react', ...)` + `await import('./sentry')` after the mock, so the dynamic import sees the mocked module.
- **[pattern]** To verify a newly-added `VITE_*` env var actually reached a preview build (Vite bakes env at build time): (1) `vercel ls --meta githubCommitSha=<sha>` to find the current preview URL for the PR's HEAD commit, (2) `vercel redeploy <url>` to force a rebuild against the new env, (3) `vercel curl <new-preview>/` to fetch the HTML and grep for the `assets/*.js` bundle, (4) `vercel curl <new-preview>/assets/<bundle>.js | grep <expected-value>` to confirm. `vercel curl` reuses the linked CLI session for deployment-protection auth.
- **[pattern]** To force a serverless 5xx on preview without touching prod: `vercel env add <KEY> preview <branch> --value INVALID --yes` (per-branch override), `vercel redeploy <previous-preview-url>` to rebuild against the override, exercise the endpoint, then `vercel env rm <KEY> preview <branch> --yes` to restore. Per-branch overrides shadow the broader value for just that branch.
- **[pattern]** Sentry MCP can be added on the fly with `claude mcp add --transport http sentry https://mcp.sentry.dev/mcp`. After adding, the `mcp__sentry__*` tools appear in the deferred-tools list — `ToolSearch` with `select:mcp__sentry__whoami,mcp__sentry__find_organizations,mcp__sentry__search_issues` (etc.) to load schemas. For this account (`you@example.com`) the org slug is `solo-23`, `regionUrl` is `https://us.sentry.io`, and the Sentry project slug for this repo is `fog_and_frontier` — pass these to every Sentry MCP call.
- **[gotcha]** `mcp__sentry__search_issues` with a narrow `firstSeen:-1h` window plus free-text query terms can return no results even when the event exists — the free-text matcher is unreliable for ephemeral test-event titles. Drop the free-text, widen the window (`firstSeen:-2h`), and add `projectSlugOrId: "<slug>"` + `sort: "new"` to surface recent events.

The user must accept terms in their browser at the printed `verification_uri` (or run `vercel integration accept-terms <product> --yes` after reading policy links), then retry `vercel integration add <product>`. - **[gotcha]** A Vercel preview build for this repo can fail with `error TS2550: Property 'cause' does not exist on type 'Error'. Try changing the 'lib' compiler option to 'es2022' or later.` even though `tsconfig.api.json` already sets `target: es2023, lib: ["ES2023"]` and the same `tsc -b` succeeds locally with the same TypeScript version. Vercel's tsc 6.0.3 resolution requires the explicit `"ES2022"` entry in `lib` to pick up `Error.cause`; `["ES2023"]` alone is not enough. Fix: `"lib": ["ES2022", "ES2023"]`. - **[pattern]** Sentry React Router v7 browser tracing wiring: pass `{ useEffect, useLocation, useNavigationType, createRoutesFromChildren, matchRoutes }` from `react-router-dom` to `Sentry.reactRouterV7BrowserTracingIntegration` inside `Sentry.init`'s `integrations: []` array. Then at module scope in `App.tsx`, wrap `Routes` once: `const SentryRoutes = Sentry.withSentryReactRouterV7Routing(Routes);` and use `<SentryRoutes>…</SentryRoutes>` instead of `<Routes>`. This produces parameterised span names (e.g. `/explore`) rather than concrete URLs. Use `tracesSampleRate: environment === 'production' ? 0.1 : 1.0` to stay inside Sentry's free-tier quota while keeping preview/dev fully sampled.
- **[gotcha]** `vi.fn(() => value)` inside `vi.hoisted(...)` without an explicit type parameter creates a mock whose `.mock.calls` is typed as an array of empty tuples, so `mock.calls[0]?.[0]` errors with `TS2493: Tuple type '[]' of length '0' has no element at index '0'`. Fix: provide the function signature as a type parameter — `vi.fn<(opts: Record<string, unknown>) => { name: string }>(() => ({ name: '…' }))`. Now `.mock.calls[0]?.[0]` is correctly typed without needing a cast (which would also trip `no-unnecessary-type-assertion`).
- **[gotcha]** When `vi.hoisted` returns multiple mocks (e.g. `{ init, reactRouterV7BrowserTracingIntegration }`), `beforeEach(() => { init.mockClear() })` is not enough — every mock has its own call-count state and accumulates across tests. Symptom: `expected "vi.fn()" to be called 1 times, but got 4 times` (one per prior test that exercised the same path). Fix: clear all hoisted mocks in `beforeEach`.
- **[gotcha]** After a force-push, `vercel ls --meta githubCommitSha=<new-sha>` can return `No deployments found under <team>` for several minutes even though the PR's status checks show a successful Vercel build for that commit. The truth is in `gh pr view <N> --json statusCheckRollup` — the `Vercel` StatusContext entry's `targetUrl` is the inspector URL for the new deploy; pair with `vercel ls` (no meta filter) to see the preview URL itself by age.
- **[gotcha]** `@typescript-eslint/no-unnecessary-type-assertion` fires on `value as T`, assertion-at-declaration patterns (`const req = fakeReq() as VercelRequest & {...}`), `(import.meta.env.MODE as string)` (Vite already types MODE as string), `as` casts inside inline object literals when the receiver already accepts that type (e.g. `[{...} as Exception]` when array element type is already `Exception`), AND `out.contexts?.trace?.data as Record<string, unknown> | undefined` when TS has already inferred a compatible type. Fix: declare a typed const first (`const exc: Exception = {...}`) and reference it; or remove the cast and let inference do its job.
- **[gotcha]** `@sentry/react` does NOT re-export `TransactionEvent` (it does re-export `Breadcrumb`, `ErrorEvent`, `Exception`). Trying `import type { TransactionEvent } from '@sentry/react'` fails with `TS2305: Module '"@sentry/react"' has no exported member 'TransactionEvent'` and `TS2694: Namespace '…@sentry/react…' has no exported member 'TransactionEvent'` (for `Sentry.TransactionEvent` via the namespace import). Fix: `import type { TransactionEvent } from '@sentry/core'` — `@sentry/core` is the canonical export point. Same pattern likely applies to other transaction/span-side types.
- **[gotcha]** Span `data` on Sentry transaction events is typed as `SpanAttributes` (a structured partial type), NOT `Record<string, unknown>`. A helper typed as `(data: Record<string, unknown> | undefined) => Record<string, unknown> | undefined` cannot have its return assigned back to `span.data` — TS errors with `Type 'Record<string, unknown> | undefined' is not assignable to type 'SpanAttributes'`. Fix: write the helper as a generic `<T extends object>(data: T): T` so the input type round-trips through the function and assignment back to `span.data` / `trace.data` typechecks cleanly. Bonus: removes the need for inner `as Record<string, unknown>` casts that `no-unnecessary-type-assertion` would otherwise flag.
- **[pattern]** When adding Sentry browser tracing, set `tracePropagationTargets: [/^\//]` in `Sentry.init` to restrict distributed-trace headers (`sentry-trace`, `baggage`) to same-origin requests. The default (`['localhost', /^\//]`) will start propagating those headers to any third-party fetch added later (Clerk, Turso, Gemini, etc.), which leaks internal trace IDs across vendor boundaries. Pair with `beforeSendTransaction` to scrub URL-shaped attributes (`http.url`, `url`) on root-trace `contexts.trace.data` AND on each span's `data` — error-event scrubbing alone is not enough once tracing is on.

- **[gotcha]** This repo squash-merges PRs (the merge commit on `main` has a fresh SHA, e.g. `72c7a08 feat(observability): ... (#20) (#47)`), so `git merge-base --is-ancestor <branch> origin/main` returns exit 1 ("not merged") even when the PR has clearly landed. To safely decide whether a local branch can be deleted, use `gh pr view <N> --json state,mergeCommit,mergedAt` (look for `state: "MERGED"`) and/or `git log origin/main --oneline --grep="<distinctive phrase from branch>"` to find the squash commit — don't trust `is-ancestor`.
- **[pattern]** To recover context after a session crash / laptop restart, prior conversation transcripts live in `/Users/<user>/.claude/projects/-Users-<user>-Code-<Repo>/*.jsonl`. `ls -lt` the directory to find the most recent one, then `grep -li "<keyword>" *.jsonl` across all of them to surface which session discussed a given topic. Also check `.caliber/learning/current-session.jsonl` for the post-tool-use stream of the in-progress session. Queued user prompts (typed but not yet processed when the session died) appear as `type: "queue-operation", operation: "enqueue"` entries in the jsonl — these are the truest record of "what the user asked next."
